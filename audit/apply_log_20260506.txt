================================
KNOWN RESIDUALS (non-blocker)
================================
[a] Views retain REFERENCES + TRIGGER privileges on anon/authenticated.
    Cosmetic noise; can be cleaned up in a v1.1 hygiene pass alongside
    the augl_id back-link concern (PLANNING_BACKLOG follow-up).

[b] Postgres 15+ view security_invoker default — by default, views run
    with the privileges of the view OWNER (typically postgres
    superuser), which means a view CAN bypass RLS on underlying tables
    if not explicitly configured with security_invoker=true. This is
    NOT a current security hole because there are zero
    service-role-only tables in the post-fix state, so no view can
    "leak" data anon shouldn't see. But this becomes critical the
    moment Áfangi 0 introduces rejected_commercial_listings as a
    service-role-only table — any view that joins that table must
    declare security_invoker=true. Logged in dry-run plan addendum and
    will be added to Áfangi 0 implementation checklist.

[c] augl_id back-link concern on last_listing_text and
    properties.augl_id_latest — exposing per-source listing IDs to
    anon enables third-party scrapers to back-link our data to source
    platforms. Not in scope of this RLS audit (alert was about
    RLS-disabled, not column exposure). Filed as PLANNING_BACKLOG v1.1
    hardening item: column-stripping view-pattern post-Áfangi-0.
