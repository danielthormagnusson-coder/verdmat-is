## 5. LÆRDÓMUR — hvernig þetta gerðist + verklagsregla

**Hvernig:** snapshot-/vinnutöflur voru búnar til beint (`CREATE TABLE ... AS`,
handvirkt eða úr fix-scriptum) og ERFÐU default `public`-grants (arwdDxtm á anon+auth)
án þess að RLS væri virkjað. RLS-verklagið („nýjar töflur fá public_read + SELECT-only")
náði aðeins til app-birtra dashboard-taflna — snapshot-/staging-töflur runnu framhjá
því þær voru „bara tímabundnar" og enginn hugsaði um þær sem API-berskjaldaðar. En
allt í `public` er sjálfkrafa PostgREST-berskjaldað.

**Verklagsregla (fer í `app/CLAUDE.md`):** HVER ný tafla í `public` fær RLS +
þröng grants Í SÖMU migration og hún er búin til — **líka snapshot-/staging-/vinnutöflur**.
Ef taflan er ekki app-lesin: `ENABLE ROW LEVEL SECURITY` + `REVOKE ALL FROM anon,
authenticated` (default-deny, engin policy, service_role dugar fyrir ops). Ef hún ER
app-lesin: bæta við `public_read`-policy eins og áður. Aldrei skilja töflu eftir í
`public` án annaðhvort RLS-flaggs eða meðvitaðrar known-accepted-skráningar.
